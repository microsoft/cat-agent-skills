# Memory and learning

Load this after substantial work has finished, to decide whether anything reusable came out of it. Do not force this after a trivial task, and do not claim persistent memory this Skill doesn't actually have. What it can do is produce a structured artefact that a person can save wherever the organisation actually keeps such things (a knowledge base, a shared drive, a wiki).

## Questions to ask

- What worked, and what didn't?
- What would you do differently next time?
- Is this likely to recur?
- Did the work produce a genuinely reusable method, template, or decision rule?
- Did it reveal an exception pattern worth documenting?
- Should an existing process or Company DNA entry be updated as a result?
- Is this a candidate for a reusable Skill? If it looks like a strong one, go to `skill-generator.md`.

## Maturity of any proposed artefact

- **Draft** - based on limited evidence, one instance.
- **Validated** - reviewed or confirmed by someone with authority to do so.
- **Established** - used repeatedly and holding up.
- **Deprecated** - no longer recommended.

Don't promote a single successful instance straight to "established". One good outcome is evidence, not proof.

## Possible artefacts

Playbook, SOP, checklist, template, decision rule, FAQ entry, lessons-learned record, process update, Company DNA update proposal, new Skill candidate, or an improvement to an existing Skill (see `skill-generator.md`).

## Output shape

`assets/templates/lessons-learned.md` covers the general case. If the artefact is process-shaped, use `assets/templates/sop-and-process.md`; if it's a Skill candidate, use `assets/templates/skill-candidate-assessment.md`.
