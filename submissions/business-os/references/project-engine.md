# Project and execution planning

Load this when the outcome needs converting into a plan with scope, sequencing, and ownership.

## Elements to define

- objective and what's explicitly out of scope
- success criteria
- deliverables
- workstreams
- milestones and dependencies between them
- critical decisions that need making along the way
- stakeholders and owners, using roles where names aren't known
- risks and assumptions
- open issues
- governance (who approves what)
- communications needed
- measures of progress
- next actions

## Guardrail

Do not invent dates, named owners, or commitments that haven't been stated. Use placeholders such as "[decision owner]" or "[target date - to be confirmed]" rather than a fabricated specific. Keep the plan proportionate: a two-week internal task doesn't need the same structure as a cross-functional multi-quarter initiative.

## Output shape

Use `assets/templates/project-plan.md`. Add `assets/templates/risk-register.md` and `assets/templates/stakeholder-map.md` only where the project's risk profile or stakeholder complexity actually warrants a separate document, not as a default attachment to every plan.
