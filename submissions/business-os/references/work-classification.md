# Work classification, sizing and Figure It Out

Load this when a request is ambiguous, when you need to decide how much reasoning a piece of work deserves, or when the user hands you something messy such as "figure this out" or "sort this out".

## Step 1: name the outcome

Before classifying anything, state in one line what a successful result would look like. If the stated request looks like a proxy for a bigger need (for example "rewrite this paragraph" inside a document that's actually trying to win an argument, or "fix the report" when the real issue is a broken process feeding it), name the bigger need too, but do not manufacture one that isn't there.

## Step 2: classify the work

Pick a primary type and, if relevant, one or two secondary types.

- **Question** - research, explain, synthesise, answer.
- **Task** - a bounded piece of work to complete.
- **Decision** - a choice between options is required. Use `decision-engine.md`.
- **Problem** - something isn't working and the cause is unclear. Use `problem-solving.md`.
- **Project** - multi-step work needing scope, milestones, and owners. Use `project-engine.md`.
- **Incident** - something has broken or gone wrong right now. Use `incident-engine.md`.
- **Process** - how work happens needs designing or improving. Use `process-engine.md`.
- **Idea/opportunity** - explore desirability, feasibility, viability, and the next experiment.
- **Document/deliverable** - purpose, audience, evidence, structure, then create or review.
- **Research** - define the question, evidence needed, source quality, synthesis, and remaining uncertainty.
- **Communication** - a message or update is the deliverable. Use `communication-engine.md`.
- **Review/assurance** - criteria, inspection, issues, confidence, remediation.
- **Strategy** - objective, current state, options, trade-offs, capability, risk, roadmap.
- **Unknown/ambiguous** - investigate just enough to classify it, then proceed.

A request such as "our biggest customer is threatening to leave, what do we do?" is commonly Problem + Decision + Communication together. Name the primary type and treat the rest as supporting threads rather than running every methodology in full.

## Step 3: size the response

Judge these dimensions, roughly, not with false precision:

- reversibility
- financial, legal or regulatory exposure
- security or privacy exposure
- number and seniority of stakeholders
- strategic significance
- urgency
- how much depends on this being right

Then choose:

- **Fast path** - low stakes, low ambiguity, obviously reversible. Just do the task well. No framework, no plan, no risk section.
- **Structured path** - moderate complexity. A short plan, a template, or one supporting reference is enough.
- **Deep path** - high impact, ambiguous, regulated, or multi-stage. Bring in the relevant engine(s) and lenses, and be explicit about gaps, risk, and what needs sign-off.

Getting this wrong in either direction is the main failure mode: giving a shallow answer to a consequential problem, or turning "write a meeting agenda" into a project plan.

## Step 4: gaps

Sort missing information into:

- **Blocking** - cannot responsibly proceed without it. Ask.
- **Important** - proceed on a stated assumption, flagged clearly.
- **Optional** - nice to have, doesn't change the approach.

Ask a question only when the answer would change what you do next. Otherwise, state the assumption and move on.

## Figure It Out mode

This is the pattern for messy, under-specified requests: "figure out how we should do this", "this process is a mess", "something is wrong with retention", "can you sort this out".

1. Work out the likely outcome.
2. Classify the work (Step 2).
3. Size it (Step 3).
4. Gather whatever context is already available in the conversation or attached material.
5. Identify blocking gaps only.
6. Pick the supporting reference(s) that fit.
7. Go as far as you responsibly can.
8. Ask only the high-value questions identified in Step 4.
9. Produce something usable, not a description of what you would produce.
10. State next actions.
11. Verify against the Step 1 outcome where possible.
12. Note anything worth capturing as reusable learning (see `memory-and-learning.md`), but only if it's genuinely repeatable.

The user should not need to know which framework applies. That's this Skill's job, not theirs.
